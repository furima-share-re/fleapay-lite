# Documentation（説明・背景）

このディレクトリには、プロジェクトの **説明・背景・コンテキスト** を記載します。

## ⚠️ 重要な制約

このディレクトリは **正（Source of Truth）ではありません**。

### ❌ 書いてはいけないもの

- 数値（例：「7日継続率は40%」）
- 条件（例：「3回以上投稿したユーザーは」）
- 制約（例：「タイムアウトは5秒」）
- 閾値（例：「100件を超えたら」）

これらは **すべて `truth/**` に記載**してください。

### ✅ 書いてよいもの

- 背景・理由（「なぜこの機能が必要か」）
- コンセプト（「このサービスの目指す姿」）
- ユーザーストーリー（「〇〇なユーザーが〇〇したい」）
- 分析結果の解釈（数値自体は truth へ）

## ディレクトリ構成

### `strategy/`
プロジェクトの戦略・方針を記載します。

- 事業戦略
- マーケティング戦略
- 成長戦略

### `feature/`
各機能の説明を記載します。

- 機能概要
- ユーザーストーリー
- UI/UX の意図

### `analysis/`
データ分析の結果・解釈を記載します。

- ユーザー行動分析
- 市場分析
- 競合分析

**注意**: 分析で得られた数値は `truth/**` に記載してください。

### `persona/`
ターゲットユーザーのペルソナを記載します。

- ペルソナ定義
- ユーザージャーニー
- ペインポイント

## 正しい書き方の例

### ❌ 悪い例（数値が含まれている）

```markdown
# ユーザー定着戦略

7日継続率を40%にすることを目指します。
そのため、3日以内に3回以上投稿したユーザーには
プッシュ通知を送ります。
```

### ✅ 良い例（数値は truth へ）

```markdown
# ユーザー定着戦略

## 背景
ユーザーの定着率向上が最重要課題です。

## アプローチ
初期段階で投稿習慣を形成することが重要と判断しました。
そのため、一定の投稿実績を達成したユーザーには
プッシュ通知でポジティブフィードバックを提供します。

## 数値目標
→ `truth/core/kgi_kpi.yml` を参照

## 条件
→ `truth/business/phases.yml` を参照
```

## CI による検証

このディレクトリに数値・条件・制約が混入していないか、
CI により自動検証されます。

違反が検知された場合、PR はブロックされます。
